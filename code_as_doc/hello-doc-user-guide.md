# Hello Doc User Guide

Updated: 2026-03-10

This file replaces `Template_maintenance_and_using_guide.md`.
It documents the current build layout, maintenance rules, and the Git tracking rule for `docs/_build/**/**/rst/**`.

---

## 1. Source of Truth

The manual system still has three layers, and only the first two are upstream maintenance sources.

1. Template layer
   - `docs/templates/page_en/*.rst`
   - `docs/templates/page_eu/*.rst`
   - `docs/templates/page_jp/*.rst`
   - Responsibility: page structure, headings, reusable prose, and table layout

2. Data layer
   - `data/phase1/Spec_Master.csv`
   - `data/phase1/Spec_Footnotes.csv`
   - `data/phase1/spec_titles.csv`
   - `data/phase1/content_blocks.csv`
   - `data/phase1/page_registry.csv`
   - Responsibility: model-specific parameters, safety/spec content, and placeholder values

3. Materialized bundle layer
   - `docs/_build/<model>/<region>/rst/**`
   - `docs/_build/<model>/<region>/html/**`
   - `docs/_build/<model>/<region>/word/**`
   - `docs/_build/<model>/<region>/pdf/**`
   - `docs/index.rst`
   - Responsibility: generated bundle plus final outputs

Rules:

- Edit templates when structure or prose changes.
- Edit CSV when product parameters change.
- Treat `docs/_build/...` as generated output even if it is committed to Git for review.

## 2. Current Build Pipeline

The cross-platform entrypoint is `build.py`.
It wraps `tools/build_docs.py`, which still drives the actual build logic.

Current flow:

1. `python build.py rst|html|word|pdf|all`
2. `tools/build_docs.py` validates config and layout params
3. target `model` and `region` are resolved from CLI or `build.targets`
4. `product_name` is resolved from `Spec_Master.csv`
5. CSV-backed pages are generated by `tools/phase1_build.py`
6. `tools/gen_index_bundle.py` materializes the bundle
7. the bundle is written to `docs/_build/<model>/<region>/rst/`
8. `docs/index.rst` is refreshed to point at all existing bundle roots
9. `html`, `word`, and `pdf` outputs are built from the bundle when requested

Important:

- `python build.py rst` only materializes the RST bundle.
- `python build.py word`, `python build.py html`, and `python build.py pdf` all prepare the RST bundle first.
- `python build.py all` runs `html`, `word`, and `pdf` after the same prepare step.

## 3. Materialized Bundle Layout

For a target such as `JE-1000F / US`, the working bundle now lives here:

- `docs/_build/JE-1000F/US/rst/index.rst`
- `docs/_build/JE-1000F/US/rst/page/*.rst`
- `docs/_build/JE-1000F/US/rst/generated/JE-1000F/*.rst`
- `docs/_build/JE-1000F/US/rst/conf.py`
- `docs/_build/JE-1000F/US/rst/conf_base.py`
- `docs/_build/JE-1000F/US/rst/_static/**`
- `docs/_build/JE-1000F/US/rst/renderers/**`

This is the generated bundle consumed by Sphinx, HTML export, Word export, and PDF export.
It is not the canonical authoring source, but it is now a valid Git tracking surface.

---

## 4. Git Tracking Rule for RST Bundles

The `.gitignore` rule has been changed so that:

- `docs/_build/**/**/rst/**` is no longer ignored
- sibling outputs such as `docs/_build/**/**/html/**`, `word/**`, and `pdf/**` remain build artifacts

This gives you two benefits:

1. You can commit generated RST bundles per target and keep reviewable history.
2. You can export Git diffs for a single model or region as CSV and HTML reports.

What this does not change:

- `_build/.../rst/**` is still regenerated on the next build.
- If you need a durable content change, apply it upstream in templates or CSV, then rebuild.

Recommended use:

1. Build the target bundle with `python build.py rst --config ...`
2. Inspect `docs/_build/<model>/<region>/rst/**`
3. Commit the resulting RST bundle
4. Use `python build.py diff-report ...` when you need a table-style change export

## 5. Which Files You Should Edit

Edit these when the manual wording or structure changes:

- `docs/templates/page_en/*.rst`
- `docs/templates/page_eu/*.rst`
- `docs/templates/page_jp/*.rst`

Edit these when safety/spec parameters change:

- `data/phase1/content_blocks.csv`
- `data/phase1/Spec_Master.csv`
- `data/phase1/Spec_Footnotes.csv`
- `data/phase1/spec_titles.csv`

Do not use these as the primary authoring source:

- `docs/_build/<model>/<region>/rst/page/*.rst`
- `docs/_build/<model>/<region>/rst/generated/<model>/*.rst`
- `docs/_build/<model>/<region>/rst/index.rst`
- `docs/index.rst`

You may commit them for version tracking, but routine maintenance should still happen upstream.

## 6. How Safety and Spec Pages Work

Safety content is generated by `tools/phase1_build.py`.

Primary inputs:

- `data/phase1/page_registry.csv`
- `data/phase1/content_blocks.csv`
- optional page override CSV such as `data/phase1/<page_id>_blocks.csv`

Generated bundle output:

- `docs/_build/<model>/<region>/rst/generated/<model>/safety_<lang>.rst`
- materialized page include: `docs/_build/<model>/<region>/rst/page/safety_<lang>.rst`

Spec content is generated from:

- `data/phase1/Spec_Master.csv`
- optional `data/phase1/Spec_Footnotes.csv`
- optional `data/phase1/spec_titles.csv`

Generated bundle output:

- `docs/_build/<model>/<region>/rst/generated/<model>/spec_<lang>.rst`
- materialized page include: `docs/_build/<model>/<region>/rst/page/spec_<lang>.rst`

`Spec_Master.csv` remains the main source of truth for spec sections, rows, and `tpl_*` placeholders.

## 7. Placeholder Rules

Core placeholders resolved from `Spec_Master.csv`:

- `|PRODUCT_NAME|`
- `|PRODUCT_NAME_BOLD|`
- `|PRODUCT_SHORT_NAME|`
- `|PRODUCT_SHORT_NAME_BOLD|`
- `|MODEL_NO|`

Resolution source:

- `product_name` comes from `Row_key=product_name`
- `model_no` comes from `Row_key=model_no`
- `PRODUCT_SHORT_NAME` is derived from `PRODUCT_NAME`

Any `Row_key` beginning with `tpl_` becomes a placeholder.

Examples:

- `tpl_main_power_button_label` -> `|MAIN_POWER_BUTTON_LABEL|`
- `tpl_side_ac_input_spec` -> `|SIDE_AC_INPUT_SPEC|`
- `tpl_battery_pack_name` -> `|BATTERY_PACK_NAME|`

Derived behavior:

- non-empty placeholders also get `..._BOLD`
- placeholders ending in `_LABEL` also get `..._LOWER`
- multi-line `tpl_*` rows produce suffixed placeholders such as `|EXAMPLE_KEY_2|`

```powershell

python build.py html
python build.py word
python build.py pdf
python build.py all
```

```powershell
python build.py rst --config config.ja.yaml
```

## 9. Version Tracking and Diff Export

Because `docs/_build/**/**/rst/**` is now trackable, you can keep RST history per target.

Example baseline:

```powershell
python build.py rst --config config.je1000f.us.yaml --model JE-1000F --region US
git add docs/_build/JE-1000F/US/rst
git commit -m "Add JE-1000F US rst baseline"
```

Example report export for one model:

```powershell
python build.py diff-report --tracked-root docs/_build/JE-1000F --from-ref HEAD~1 --to-ref HEAD
```

Example report export for one region:

```powershell
python build.py diff-report --tracked-root docs/_build/JE-1000F/JP --from-ref HEAD~3 --to-ref HEAD
```

Default outputs:

- `reports/version_tracking/JE-1000F/*.csv`
- `reports/version_tracking/JE-1000F/*.html`

Use `--report-dir` if you want a different output folder.

## 10. Common Pitfalls

### 10.1 Editing only the tracked RST bundle

This is the main risk after enabling Git tracking on `_build/.../rst/**`.

If you change only:

- `docs/_build/<model>/<region>/rst/page/*.rst`
- `docs/_build/<model>/<region>/rst/generated/<model>/*.rst`

those changes can be overwritten on the next rebuild.

If the change should survive rebuilds, patch templates or CSV first.

### 10.2 `?` appears in output

This is usually caused by dirty `tpl_*` values in `Spec_Master.csv`, not by the template structure itself.

### 10.3 Old model names survive in the new manual

This usually means one of these happened:

- a template still contains hard-coded model text
- `product_name` in `Spec_Master.csv` was not updated
- the wrong `config`, `model`, or `region` was used

### 10.4 Hard-coded title in config

If `build.word_title` is fixed to an old model name, the generated Word title will stay wrong even if `PRODUCT_NAME` is correct.
Prefer a placeholder-based title such as:

```yaml
word_title: "|PRODUCT_NAME| User Manual"
```

## 11. Verification Checklist

After changing templates or CSV values, verify at least the following:

1. `python build.py rst --config ...` succeeds
2. the target bundle appears under `docs/_build/<model>/<region>/rst/`
3. generated pages contain no unresolved placeholders such as `|PRODUCT_NAME|`
4. generated pages contain no stale model names from older products
5. safety and spec still come from CSV-backed generated pages
6. the expected `.docx`, `.html`, or `.pdf` file is generated when requested

Useful checks:

```powershell
Select-String -Path docs\_build\JE-1000F\US\rst\page\*.rst -Pattern '\|[A-Z0-9_]+\|'
Select-String -Path docs\_build\JE-1000F\US\rst\page\*.rst -Pattern '\?'
git status --short -- docs/_build/JE-1000F/US/rst
```

---

## 12. One-Sentence Rule

Templates define structure.
CSV defines model-specific values.
`docs/_build/**/**/rst/**` is a Git-trackable generated snapshot, not the primary maintenance source.
