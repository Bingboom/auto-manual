# 📘 文档生产系统构建说明

（从单页构建到整本多语言说明书自动化）

---

# 一、当前系统能力（现状）

当前资料开发仍采用传统方式：

* 内容由资料整理
* 设计参与版式合成
* 终版确认通过协作完成
* 最终输出印刷级 PDF

在此基础上，目前已经增加以下能力：

```
CSV（结构化内容）
    ↓
phase1_build.py
    ↓
RST
    ↓
Sphinx
    ↓
LaTeX
    ↓
XeLaTeX
    ↓
单页 PDF（Safety）
```

当前增强能力特点：

* 内容结构化存储
* 排版由统一模板控制
* 内容与版式分离
* 可重复构建
* 不改变现有协作角色

这属于对传统流程的“结构化增强”，而不是流程替代。

---

# 二、下一阶段目标（在现有模式下增强能力）

🎯 目标

在保持现有资料协作方式不变的前提下，实现：

* 自动构建整本多语言说明书
* 保持设计视觉页面的独立性
* 减少重复排版工作
* 提升版本稳定性

输出形式：

* manual_en.pdf
* manual_fr.pdf
* manual_es.pdf

必须满足：

* 一条命令可构建整本
* 页面结构清晰
* 多语言内容可筛选
* 图页可插入
* 不影响现有设计协作方式

# 三、整本说明书的结构增强模型

整本说明书仍然遵循传统结构：

但在生产方式上分为两类页面。

---

## 1️⃣ 结构化内容页（增强层生成）

例如：

* Safety
* LCD Display
* Specifications
* Troubleshooting
* Warranty
* Operations
* Storage

特点：

* 内容更新频繁
* 多语言差异明显
* 结构稳定
* 逻辑固定

输入来源：

* 钉钉多维表（内容维护）
* 导出为 CSV 快照
* 进入构建流程

构建方式：

```
CSV
 ↓
phase1_build.py
 ↓
RST
 ↓
Sphinx
 ↓
LaTeX
 ↓
PDF
```

这部分是对传统内容整理工作的增强。

---

## 2️⃣ 整页视觉页（保持设计主导）

例如：

* 封面
* PRODUCT OVERVIEW
* UPS 图页
* Charging 图页
* Battery Pack 示意页

特点：

* 视觉主导
* 版式复杂
* 改动频率低
* 设计资产稳定

输入资源：

```
assets/cover_en.pdf
assets/overview_en.pdf
assets/ups_en.pdf
assets/charging_en.pdf
```

构建方式：

```
\includepdf[fitpaper=true]{overview_en.pdf}
```

说明：

* 视觉页仍由设计提供
* 不参与自动排版逻辑
* 不改变现有设计职责

---

# 四、内容管理与构建分层

为避免构建系统直接依赖线上数据，采用分层管理方式：

---

## 1️⃣ 协作层（线上维护）

工具：

* 钉钉多维表

用途：

* 内容录入
* 多语言维护
* 模块管理
* 图片关联
* 协作确认

这是日常开发层。

---

## 2️⃣ 构建快照层（导出数据）

形式：

* 从多维表导出 CSV
* 固定字段顺序
* 固定筛选规则
* 固定语言

特点：

* 作为构建输入
* 可归档
* 可回溯
* 不直接依赖线上表

构建系统只读取此层数据。

---

## 3️⃣ 构建层（Pipeline）

逻辑：

```
CSV 快照
 ↓
生成 RST
 ↓
Sphinx
 ↓
LaTeX
 ↓
XeLaTeX
 ↓
整本 PDF
```

特点：

* 离线可构建
* 可重复生成
* 与协作系统解耦

---

# 五、整本说明书构建逻辑

整本说明书由：

> 页面序列 + 构建规则

组成。

示例结构：

```
Manual
 ├── Cover (PDF)
 ├── Intro (CSV)
 ├── TOC (自动生成)
 ├── Safety (CSV)
 ├── Symbols (CSV)
 ├── Overview (PDF)
 ├── LCD Display (CSV)
 ├── Operations (CSV)
 ├── UPS (PDF + CSV)
 ├── Charging (PDF + CSV)
 ├── Troubleshooting (CSV)
 ├── Specifications (CSV)
 ├── Warranty (CSV)
 └── App Setup (CSV)
```

页面类型分为：

| 页面类型 | 构建方式     |
| ---- | -------- |
| 视觉页  | 插入设计 PDF |
| 内容页  | CSV 自动生成 |

---

# 六、多语言处理方式（增强模式）

多语言仍在协作层维护。

构建时：

* 导出对应语言 CSV
* 构建对应语言 PDF

语言成为：

> 数据维度，而不是文件维度。

不改变现有翻译流程，只改变生成方式。

# 八、0 → 1 阶段完成标志

在不改变传统资料开发模式的情况下，实现：

# 九、本质变化（增强版表达）

传统方式：

* 文档 = Word 文件
* 多语言分文件
* 人工排版

增强方式：

* 文档 = 数据 + 模板 + 页面规则
* 多语言 = 数据筛选
* 构建 = 可重复过程

不改变协作结构，只减少重复劳动。

---

# 十、当前阶段优先级

当前重点：

* 梳理页面结构
* 分类页面类型
* 固化构建顺序
* 稳定构建流程

暂不追求：

* 全自动视觉页
* 复杂图页参数化
* 完全自动翻译

---

# 总结（现实定位）

当前系统不是革命。

而是：

> 在传统资料开发流程下，引入结构化构建能力。

目标是：

* 降低重复排版工作
* 提高版本稳定性
* 提升构建效率
* 为未来自动化留出接口
