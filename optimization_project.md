# Optimization Project

Updated: 2026-03-10

## 1. Background

This repository already supports:

- cross-platform build entrypoint via `python build.py rst|word|html|pdf|all`
- target-scoped outputs under `docs/_build/<model>/<region>/`
- target coexistence across multiple configs such as `config.yaml` and `config.ja.yaml`
- Git tracking for `docs/_build/**/**/rst/**`
- CSV/HTML diff export via `python build.py diff-report`

The current system is usable, but the next bottlenecks are now clear:

- `_build/<model>/<region>/rst/**` mixes runtime build assets and reviewable text files
- quality checks are still fragmented
- template dependencies on `tpl_*` and spec rows are implicit
- diff reports are file-level only
- release traceability is incomplete

This optimization plan is based on the current repository structure and on ideas borrowed from the ESP-Docs user guide, especially:

- multi-target documentation from one source
- local preview and fast build workflows
- clear build, check, troubleshooting, and release layers
- CI-friendly validation gates

---

## 2. Optimization Goals

The next stage should move the repo from "can build manuals" to "can reliably operate a documentation product line".

Primary goals:

1. Separate build runtime artifacts from Git review artifacts.
2. Make template and data dependencies explicit before build.
3. Add one quality gate that can be used locally and in CI.
4. Improve version tracking from file-level diff to page-level and field-level diff.
5. Add release metadata so every output can be traced back to config, spec source, and Git commit.

## 4. Guiding Principles

1. Generated files may be committed for review, but they are still not the upstream maintenance source.
2. Runtime build layout and review layout should not be the same thing.
3. Every page should declare what data it requires.
4. Every build should produce enough metadata to explain itself later.
5. Every CI gate should be runnable locally with the same command.

## 6. Phase Plan

### Phase 1: Review Bundle Separation

Goal:

- stop using full runtime `_build/.../rst/**` as the main Git review surface

Implementation:

1. Extend [tools/gen_index_bundle.py](C:\Users\Administrator\Documents\GitHub\CMS\auto-manual\tools\gen_index_bundle.py) to optionally emit a review bundle.
2. Add a new helper module such as `tools/review_bundle.py`.
3. Copy only:
   - `index.rst`
   - `page/*.rst`
   - `generated/<model>/*.rst`
4. Generate `manifest.json` alongside the review bundle.
5. Switch `.gitignore` and diff-report defaults from `_build/.../rst` to `_review`.

Command impact:

- `python build.py rst` still builds runtime bundle
- new `python build.py review` generates review bundle
- `python build.py all` may optionally generate review bundle automatically

Expected benefit:

- smaller Git noise
- cleaner diff reports
- less confusion about what should be reviewed vs what only exists for Sphinx

### Phase 2: Unified Quality Gate

Goal:

- one local and CI entrypoint for all core validation

Implementation:

Add a new action to [build.py](C:\Users\Administrator\Documents\GitHub\CMS\auto-manual\build.py):

- `python build.py check`

The check pipeline should include:

1. config validation
2. layout params validation
3. required `product_name` and `model_no` validation
4. unresolved placeholder scan
5. stale model-name scan
6. missing referenced asset scan
7. broken local include/reference scan
8. required spec/template key validation

Potential module split:

- `tools/check_build_inputs.py`
- `tools/check_bundle_outputs.py`
- `tools/check_placeholders.py`

Expected benefit:

- local feedback before Word/PDF export
- CI and local parity
- fewer late-stage failures

### Phase 3: Page Contracts

Goal:

- make template dependencies explicit

Implementation:

Introduce contract files such as:

- `docs/templates/contracts/03_product_overview.yaml`
- `docs/templates/contracts/05_operation_guide.yaml`

Contract schema should declare:

- page id
- required `tpl_*` keys
- required spec row keys
- required assets
- allowed languages
- allowed regions
- allowed models if needed

Validation integration point:

- before `_materialize_planned_page` in [tools/gen_index_bundle.py](C:\Users\Administrator\Documents\GitHub\CMS\auto-manual\tools\gen_index_bundle.py)

Expected benefit:

- clear pre-build failures
- better onboarding for new products
- less manual template archaeology

### Phase 4: Multi-Target Content Strategy

Goal:

- reduce page duplication across US, JP, EU, and future regions

Implementation direction:

1. Keep one canonical page template when structure is shared.
2. Introduce block-level condition mechanisms:
   - `region_only`
   - `lang_only`
   - `model_only`
   - `feature_flag`
3. Prefer small conditional blocks over full page forks.

Possible implementation forms:

- custom RST directives
- preprocessing markers resolved before materialization
- YAML-driven page fragment assembly

Expected benefit:

- less copy-paste drift
- easier translation sync
- lower maintenance cost

### Phase 5: Diff Report Upgrade

Goal:

- move beyond file-level diff

Implementation:

Extend [tools/diff_report.py](C:\Users\Administrator\Documents\GitHub\CMS\auto-manual\tools\diff_report.py) with three report tiers:

1. file-level
2. page-level
3. field-level

Field-level should focus on:

- `tpl_*` substitutions
- `product_name`
- `model_no`
- spec row values where the row key can be parsed

Suggested outputs:

- `reports/version_tracking/<scope>/files.csv`
- `reports/version_tracking/<scope>/pages.csv`
- `reports/version_tracking/<scope>/fields.csv`
- matching HTML summaries

Expected benefit:

- product, QA, and documentation reviewers can read changes without opening raw diff

### Phase 6: Release Manifest and Traceability

Goal:

- explain every output after the fact

Implementation:

Add:

- `python build.py release-manifest`

This command should collect:

- build arguments
- resolved targets
- resolved product names
- source CSV paths
- output paths
- file hashes
- Git SHA

Expected benefit:

- easier release audits
- easier rollback analysis
- easier handoff between engineering and documentation teams

## 7. Command Roadmap

### Current commands

```powershell
python build.py rst
python build.py word
python build.py html
python build.py pdf
python build.py all
python build.py diff-report
```

### Planned commands

```powershell
python build.py review
python build.py check
python build.py preview --model JE-1000F --region US --page 05_operation_guide
python build.py fast --config config.yaml
python build.py release-manifest --config config.yaml
python build.py diff-report --level fields --tracked-root docs/_review/JE-1000F
```

Command intent:

- `review`: generate Git-friendly review bundle
- `check`: run all validations without full export
- `preview`: build only one page or a narrow subset
- `fast`: skip expensive steps when debugging text and placeholders
- `release-manifest`: emit release metadata
- `diff-report --level fields`: export field-level change table

CI principles:

- same commands locally and in CI

- explicit artifact retention for review outputs

## 9. Risks and Mitigations

### Risk 1: `_review` and `_build` diverge

Mitigation:

- always generate `_review` from the already materialized runtime bundle
- add manifest hash comparison

### Risk 2: contracts become stale

Mitigation:

- make contracts small and page-local
- validate contracts in CI
- fail fast on missing keys

### Risk 3: field-level diff becomes too heuristic

Mitigation:

- start with explicit keys only
- prefer structured extraction over raw text parsing

### Risk 4: more commands create confusion

Mitigation:

- keep `rst`, `word`, `html`, `pdf`, `all` unchanged
- add new commands without breaking current workflow
- document each command in README and `user-guide/hello_auto-doc.md`

---

## 10. Recommended Implementation Order

The recommended order is:

1. review bundle layer
2. unified `check` command
3. page contracts
4. diff-report upgrade
5. release manifest
6. preview/fast build refinements

Reason:

- steps 1 and 2 produce immediate operational value
- steps 3 and 4 reduce maintenance ambiguity
- steps 5 and 6 improve release discipline and developer experience
